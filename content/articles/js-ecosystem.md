# JavaScript 生态的疲劳与解药

2026 年了。我们有了第 18 个版本的 React，第 36 个构建工具，和第 9 个「下一代」状态管理库。一个简单的博客项目需要 1.2GB 的 `node_modules`。

这正常吗？

## 疲劳的症状

JavaScript 生态疲劳不是新鲜事。早在 2016 年，就有人用「JavaScript fatigue」来描述这种感觉：你刚学会一个工具，它就被宣布过时；你刚搭建好一套技术栈，社区已经转向了下一套。

但问题不在于「有太多选择」，而在于**选择的成本被转嫁给了每一个开发者**。

```javascript
// 这只是一个博客，为什么需要这些？
import { createServerComponentClient } from '@supabase/auth-helpers-nextjs'
import { cache } from 'react'
import { notFound } from 'next/navigation'
```

## 解药：朴素优先

一个激进但有效的原则：**永远从最简单的方案开始。** 这句话人人都会说，但很少有人真的实践。

简单意味着：

1. **无框架起步。** 一个 `index.html`、几个 CSS 文件、一段原生 JS。能走多远走多远。
2. **不到疼的时候，不要加依赖。** 不是在你能预见疼的时候——而是你真的感到疼的时候。
3. **优先使用平台。** `fetch`、CSS Grid、`<dialog>`、Web Components——现代浏览器已经提供了许多过去需要库才能实现的能力。

## 真正重要的是什么

用户不关心你用了什么框架。他们在乎的是：页面加载快不快？文字好不好读？导航直觉不直觉？

**技术选型的唯一标准是它是否服务于用户体验。** 不是为了「开发者体验」（DX），不是为了简历上的关键词，更不是为了追随 Hacker News 首页上的热门项目。

如果你用一个 300KB 的 JS 框架来渲染一段 2KB 的博客文章，你是在用技术膨胀来弥补设计上的不安全感。
